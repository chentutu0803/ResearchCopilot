# 研究副驾驶（ResearchCopilot）实施计划

## 一、项目定位

### 1. 项目名称
**研究副驾驶（ResearchCopilot）**

### 2. 一句话介绍
这是一个**工作流驱动**的深度研究助手：用户输入一个研究主题，系统自动完成**问题拆解、资料搜索、知识检索、证据整理、带引用写作、结果评审与可观测追踪**，最终输出一篇**可追溯、可验证**的研究报告。

### 3. 项目面向的岗位
该项目主要用于面试以下方向：

- 大模型开发
- Agent 应用开发
- 检索增强生成（RAG）相关工程岗位
- 智能体工作流与工具调用方向岗位

### 4. 这个项目的核心卖点
本项目的重点不是“有多少个 Agent”，而是：

1. **结果可追溯**：每个重要结论都能找到对应来源
2. **过程可观测**：每一步调用、检索、生成都有日志
3. **效果可量化**：检索和工作流设计可以做对比实验
4. **系统可解释**：为什么这么设计、每个模块解决什么问题，都能说清楚

---

## 二、项目目标与边界

### 1. 项目目标
用户输入一个研究主题后，系统应能够：

1. 将主题拆分为 3 到 5 个可研究的子问题
2. 针对每个子问题执行搜索与检索
3. 对搜索结果和知识库结果进行统一整理
4. 基于证据生成结构化研究报告
5. 在报告中标注逐句或逐段引用
6. 对初稿进行评分和审查
7. 在必要时进行有限轮次修订
8. 输出最终报告、引用列表和完整 Trace 日志

### 2. 第一版不做什么
为了保证项目能在一周内做完并且适合面试展示，以下内容不纳入第一版：

- 长期记忆
- 多智能体自由通信框架
- 代码执行沙箱
- 单独的 FastAPI 后端
- 多用户并发系统
- 完整的线上部署体系

第一版强调的是：**先把核心链路做通、做稳、做得能解释**。

## 三、核心设计思路

### 1. 为什么采用工作流，而不是完全自治的多智能体
这个项目的任务本质上是一个较强线性的流程：

**先规划，再检索，再写作，再审查。**

如果一开始就做成完全自治的多智能体系统，会带来几个问题：

- 难以控制执行路径
- 出错后难以定位问题
- 面试时很容易被问“为什么一定要这么复杂”
- 短时间内容易做成概念 Demo，而不是工程项目

因此，第一版采用**可控工作流**最合理：

- 更容易调试
- 更容易记录 Trace
- 更容易做评测和对比实验
- 更容易在面试中解释设计取舍

### 2. 什么时候升级成多智能体
后续如果要升级，可以在以下场景中引入并行研究者：

- 子问题数量很多且相互独立
- 希望提升整体研究速度
- 希望对不同子问题使用不同检索策略

但这属于第二阶段优化，不是第一版重点。

## 四、总体架构

整体流程如下：

`用户主题 -> 规划模块 -> 研究模块 -> 证据整理 -> 写作模块 -> 审查模块 -> 最终报告 + 引用列表 + Trace`

### 1. 规划模块（Planner）
职责：

- 将用户输入的研究主题拆分为 3 到 5 个子问题
- 给出建议性的研究大纲
- 尽量避免重复和过于空泛的问题

输入：

- 用户主题

输出：

- 子问题列表
- 可选的大纲结构

### 2. 研究模块（Researcher）
职责：

- 围绕单个子问题执行有上限的 ReAct 式检索循环
- 调用 Web Search 或知识库检索工具
- 判断当前证据是否足够
- 当检索失败或结果差时自动做 query rewrite

输入：

- 单个子问题
- 可用工具列表

输出：

- 证据列表
- 检索过程日志

### 3. 写作模块（Writer）
职责：

- 基于已有证据写出结构化研究报告
- 明确标注引用编号
- 限制模型只使用证据包中的内容，不允许随意扩写

输入：

- 子问题摘要
- 证据包

输出：

- 研究报告初稿

### 4. 审查模块（Reviewer）
职责：

- 对初稿进行打分
- 检查事实支撑、引用覆盖率、逻辑完整性
- 低于阈值时输出结构化反馈
- 带着反馈回到前面的规划、检索与写作阶段
- 整个闭环最多只允许有限轮次，防止无限循环

输入：

- 报告初稿
- 证据包

输出：

- 评分结果
- 审查反馈
- 是否需要重新规划/补检索/重写

### 5. 追踪模块（Tracer）
职责：

- 记录每一步执行情况
- 保存工具调用参数、输出摘要、耗时等信息
- 支持在 Streamlit 页面中展示

### 6. 评测模块（Evaluator）
职责：

- 跑检索实验
- 跑工作流消融实验
- 记录效果差异
- 为面试提供数据支撑

## 五、技术选型

| 组件 | 选型 | 选择理由 |
|---|---|---|
| 大模型 | DeepSeek V3 API | 中文表现好，成本较低，适合快速原型 |
| 向量模型 | bge-small-zh-v1.5 | 轻量、中文检索效果较好 |
| 重排序模型 | bge-reranker-v2-m3 | 面试时容易讲清精排带来的提升 |
| 向量存储 | FAISS | 本地化、启动快、依赖少 |
| 关键词检索 | rank_bm25 | 简单直接，适合做混合检索基线 |
| 网络搜索 | duckduckgo-search | 无需额外 API Key，便于快速演示 |
| 前端展示 | Streamlit | 上手快，演示效果直观 |
| 测试 | pytest | Python 生态常用、轻量 |
| 配置管理 | python-dotenv / pydantic | 环境变量管理清晰 |

## 七、功能需求说明

### 1. 输入
用户在页面输入一个研究主题，并可选择以下参数：

- 报告语言
- 子问题数量上限
- 最大搜索轮数
- 是否启用本地知识库

### 2. 输出
系统最终输出以下内容：

- 一份结构化研究报告（Markdown）
- 带编号引用的正文
- 参考资料列表
- 每一步工作流的 Trace 记录
- 评测或实验结果摘要（可选）

### 3. 系统应具备的能力

#### （1）主题规划
- 将输入主题拆分为 3 到 5 个子问题
- 子问题要具体、可搜索、可写作
- 不能出现明显重复

#### （2）检索与搜索循环
针对每个子问题，系统需要：

- 判断当前是否需要更多信息
- 调用 Web Search 或知识库检索
- 获取工具结果后判断是否足够
- 在搜索效果差时自动改写 query
- 到达上限后停止循环

#### （3）证据管理
- 将网页结果和本地知识库结果统一为同一种数据结构
- 记录来源网址、标题、摘要、块编号等元信息
- 对高度重复结果做去重

#### （4）带引用写作
- 写作时每个重要结论必须能对应到证据项
- 报告结构要清晰，适合展示
- 对证据不足的地方要允许模型明确表达“不足以支持”

#### （5）自动审查
- 从事实支撑、引用覆盖、逻辑连贯、结构完整四个维度打分
- 低于阈值时触发定向反馈，并重新进入前面阶段补证据
- 整个闭环有次数上限，不允许无限循环

#### （6）可观测性
- 记录模型输入摘要
- 记录工具调用参数和返回摘要
- 记录每步耗时
- 如有 token 信息，也应记录

#### （7）可评测性
- 支持构建小规模 eval 集
- 支持对检索方案做对比实验
- 支持对工作流设计做消融实验

## 八、知识库与数据规划

### 1. 推荐第一版的数据范围
为了提高可控性，建议第一版只做一个垂直方向的知识库，例如：

- 大模型推理优化
- RAG 系统设计
- Agent 工作流工程实践

这样做有三个好处：

1. 更容易准备高质量语料
2. 更容易设计 eval 查询
3. 更容易在面试中展示“对一个方向做深”的能力

### 2. 可选数据来源
- arXiv 论文
- 官方技术博客
- 框架文档
- 基准测试或技术综述文章

### 3. 数据处理流程
数据进入知识库前，按以下流程处理：

1. 收集原始文档
2. 统一转为文本或 Markdown
3. 按段落或语义块切分
4. 建立 BM25 索引
5. 建立向量索引
6. 给每个块附加元信息：
   - 来源标题
   - 来源网址
   - 来源类型
   - 块编号
   - 时间信息

- 指数退避重试

### 输出字段建议
- 标题
- 链接
- 摘要

### 验收标准

---

## 7. `rag/chunker.py`

### 职责
- 将原始文档切分为适合检索的文本块

### 建议策略
- 先按自然段切分
- 再按长度做必要拆分
- 对特别长的段落加入少量重叠

### 验收标准
- 文本块既不过碎，也不过长
- 每个块都能回溯到原始来源

---

## 8. `rag/retriever.py`

### 职责
- 实现混合检索
- 合并 BM25 与 dense retrieval 的候选结果

### 建议流程
1. 跑 BM25 top-k
2. 跑向量检索 top-k
3. 合并候选集合
4. 去掉重复项
5. 送入精排模块

### 验收标准
- 对测试查询返回较相关结果
- 混合检索在 eval 集上优于至少一种单路检索方案

## 9. `rag/reranker.py`

### 职责
- 对候选结果做精排

### 验收标准
- 精排后结果顺序更符合查询需求
- 单次延迟在演示场景下可接受

### 职责

### 提示词设计重点

### 验收标准

## 13. `agents/researcher.py`

### 职责
- 针对每个子问题执行受限的检索循环

### 单轮逻辑建议
1. 判断当前信息是否足够
2. 选择使用 Web Search 还是知识库工具
3. 构造检索 query
4. 执行工具调用
5. 观察结果质量
6. 决定停止、继续或重写 query

### Query Rewrite 建议策略
- 简化过长 query
- 补充领域关键词
- 展开缩写词
- 缩小过于宽泛的搜索范围

### 验收标准
- 严格遵守最大轮数上限
- 遇到空结果能自动重试
- 最终输出结构化证据包

## 14. `agents/writer.py`

### 职责
- 基于证据包生成带引用的结构化报告

### 写作约束
- 每个重要结论都要对应证据
- 不允许无依据编造
- 如果证据不足，应明确保守表达

### 验收标准
- 报告结构完整、语言通顺

## 15. `agents/reviewer.py`

### 职责

### 建议评分维度
- 事实支撑（1-5）
- 引用覆盖（1-5）
- 逻辑连贯（1-5）
- 结构完整（1-5）

### 审查策略
- 总分达到阈值则直接通过
- 否则给出审查反馈，并带着反馈重新规划、补检索、重写
- 闭环次数有限，防止反复回滚

### 验收标准
- 审查输出结构化、可执行
- 流程不会进入无限修订

## 十、评测与实验计划

### 1. 为什么必须做评测
如果没有评测，这个项目很容易被理解成“工作流包装过的 API 调用”。

有评测后，你可以明确说明：

- 为什么要做 Hybrid Retrieval
- 为什么要做 Re-rank
- 为什么要做 Query Rewrite
- 为什么 Reviewer 有价值

也就是说，评测是项目从“能跑”到“能 defend”的关键。

### 2. Eval 集设计
建议构建 20 到 50 条研究查询，类型尽量覆盖：

- 概念解释类
- 技术对比类
- 趋势综述类
- 方法总结类
- 原因分析类

### 3. 检索评测指标
建议至少实现：

- Hit@3
- MRR@5

如果时间够，可以再加：

- nDCG@5

### 4. 工作流效果指标
建议统计：

- 引用覆盖率
- 不被证据支持的结论比例
- Reviewer 触发率
- 平均搜索轮数
- 单次报告生成平均耗时

### 5. 必做消融实验

1. 纯向量检索
2. 混合检索
3. 混合检索 + 精排

如果时间允许，可再补两组：

4. 去掉 Query Rewrite
5. 去掉 Reviewer

### 6. 人工抽样评审
从最终输出里选 10 条结果做人审，评分维度：

- 事实性
- 引用正确性
- 连贯性
- 实用性

## 十五、面试答辩要点

### 1. 为什么不直接用 LangChain
- 我想自己实现核心工作流和工具调用协议，展示对底层机制的理解
- 这个项目的核心逻辑并不大，完全可以用纯 Python 写清楚
- 重框架会增加调试成本，不利于解释执行细节

### 2. 为什么不用完全自治的多智能体
- 研究任务主要是线性依赖，工作流更合适
- 工作流更可控，方便调试、评测和追踪
- 当前架构已经为并行研究者保留了扩展空间

### 3. 为什么要做 Hybrid Retrieval
- BM25 更擅长精确关键词、缩写词、专有名词
- 向量检索更擅长语义相似和表达变化
- 两者结合能兼顾准确匹配和语义召回

### 4. 为什么要做 Re-rank
- 混合召回之后结果通常较杂，需要精排提升前几条质量
- 精排是最容易做出量化提升的部分之一

### 5. 为什么 Reviewer 不是“再调一次 API”
- Reviewer 有明确评分维度和阈值
- 低于阈值才触发前面阶段的补闭环
- 最大闭环次数固定，不会无限循环

### 6. 为什么 Trace 很重要
- Agent 或工作流系统最难的是调试
- Trace 可以帮助定位是规划问题、检索问题还是写作问题
- 这也是系统工程能力的重要体现

# 项目亮点总结页

这份文档适合你在面试前快速复习，也适合发给面试官作为项目亮点说明。

---

## 一、项目一句话介绍

**ResearchCopilot 是一个可验证、可追溯、可观测的深度研究助手。**

用户输入研究主题后，系统自动完成：

**主题拆解 → 搜索与检索 → 证据整理 → 带引用写作 → 自动审查 → 输出研究报告与 Trace**

---

## 二、为什么这个项目适合面试

这个项目不只是“调 API”，而是能体现以下几类工程能力：

1. **Agent 工作流设计能力**
2. **RAG 系统实现能力**
3. **工具调用与失败重试能力**
4. **系统可观测与调试能力**
5. **评测与实验意识**

---

## 三、最值得讲的 6 个亮点

### 1. Workflow 优先，而不是盲目堆多智能体

我没有一开始把它做成完全自治多智能体系统，而是采用了：

`LangGraph StateGraph -> Planner -> Researcher -> ReAct Writer -> Reviewer`

原因是：

- 研究任务本质上是强线性依赖
- workflow 更可控
- 更容易调试
- 更容易做评测和消融实验
- LangGraph 让状态流转、回退边界和节点职责更清晰

这能体现你对“工程可控性”的理解，而不是只追求看起来更复杂的架构。

---

### 2. ReAct Writer + 带引用写作，解决“结果不可信”的问题

Writer 节点不是一次性模板拼接，而是一个会读取子问题和证据工具的 ReAct agent，最终输出的不是普通文章，而是：

- 每个关键结论都尽量对应证据项
- 报告中带引用编号
- 最终有参考资料列表

这可以直接回答面试官常问的问题：

> 你怎么降低幻觉？你怎么让输出更可信？

---

### 3. Hybrid Retrieval + Web Candidate Rerank

项目实现了：

- BM25
- Dense Retrieval
- Re-rank
- Tavily Web Search
- Web 候选结果 LLM Rerank

你可以讲清楚为什么需要两段检索：

- BM25 处理关键词、专有名词、缩写词
- Dense Retrieval 处理语义相似表达
- Re-rank 提升前几条结果质量
- Web Search 先召回候选 URL，再让 LLM 基于 `title/url/snippet` 删除无关结果并排序

这非常适合大模型开发 / RAG 岗位面试。

---

### 4. Query Rewrite 与失败重试

Researcher 不是只搜一次。

当结果为空或证据不足时，会：

- 自动改写 query
- 有上限地继续尝试
- 防止系统一次失败就退出

这个点体现的是：

**你做的不是 demo，而是更接近真实可用系统。**

---

### 5. Reviewer 有明确评分规则和退出边界

Reviewer 不是“再调一次模型”这么简单，而是：

- 有评分维度
- 有通过阈值
- 有重新规划闭环的上限

这能体现你对 Agent 系统“必须可收敛”的理解。

---

### 6. 有 Trace、有工件、有评测

这是这个项目最工程化的地方：

- 每一步有 Trace
- 每次运行会保存结构化工件
- 有检索评测和消融实验
- Web 检索链路还能单独输出候选数、LLM 筛选后 URL 数、fetch 成功数
- Writer ReAct 过程会完整保存下来，能回看每轮工具调用
- `run_summary` 能直接看到 web 检索轮数、writer react 轮数、reviewer 回退轮数

面试里你可以直接展示：

- `latest_run.json`
- `retrieval_eval_summary.json`
- `workflow_ablation_summary.json`
- `workflow_metrics_summary.json`

这比口头说“我做了 RAG 优化”更有说服力。

---

## 四、你可以怎么回答“为什么这个项目有价值”

一个很稳的回答方式：

> 这个项目的价值不在于做了多少个 Agent，而在于我把研究型 Agent 系统的几个关键问题都做了工程化处理：
> 1）结果如何追溯，
> 2）检索质量如何提升，
> 3）失败时如何恢复，
> 4）系统如何调试，
> 5）效果如何量化。

## 六、面试官最可能追问的问题

### 1. 为什么不用 LangChain？

因为这个项目的核心规模并不大，自己实现更能展示我对：

- 状态流转
- 工具调用协议
- 结构化输出
- Trace

这些底层细节的理解。

### 2. 为什么不做成完全自治多智能体？

因为研究任务更适合线性 workflow，先做可控、可测、可解释的系统，再考虑并行多智能体扩展。

### 3. Reviewer 有什么实际意义？

它不是简单重跑模型，而是先判断当前结果是否真的达标；如果不达标，就带着审查反馈回到前面的规划和检索阶段补证据，而不是只在原稿上修修补补。

### 4. 你怎么证明检索设计真的有用？

我做了检索评测和工作流消融，不只是展示最终页面，还展示指标摘要和运行工件。

### 5. 你怎么定位系统问题？

我保留了 Trace 和结构化运行结果，可以区分规划、检索、写作、审查各阶段的问题。

## 七、建议你面试时的展示顺序

1. 先打开 README，30 秒讲项目定位
2. 再打开 Streamlit 演示页面
3. 展示一条真实研究主题的输出
4. 展示 `latest_run.json`
5. 展示评测摘要与截图
6. 最后讲设计取舍和可扩展方向

## 八、最后总结

这个项目真正的价值不在“功能很多”，而在：

- 它是完整链路
- 它有工程结构
- 它有证据链
- 它有评测
- 它能解释为什么这么设计

对面试来说，这比一个“看起来很炫”的 API 套壳项目更有竞争力。

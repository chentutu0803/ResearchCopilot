# 面试提问与回答（按面试官视角整理）

## 项目简介

我的项目叫“GPU 领域双语问答模型训练与 vLLM 推理优化”。这个项目最早来自课程作业，但我在准备面试的时候，不会把它只讲成推理项目，也不会讲成特别夸张的大平台。更准确地说，它是一个从数据构造、LoRA 微调到 vLLM 推理优化都做过一轮的项目，只不过最终目标是提升离线推理吞吐。

它主要想解决的问题其实很直接：如果只做训练，不考虑部署和吞吐，那最后在评测和交付上价值有限；反过来，如果只讲推理优化，没有一个像样的领域模型基础，实验也会比较空。所以我做这个项目时，是按“数据构造 + LoRA 微调 + 权重合并 + vLLM 推理优化”这条链路来做的。具体来说，我先基于《GPU 架构与编程》教材 PDF 整理中英双语问答数据，统一成 ShareGPT(messages) 格式，再基于 Qwen2.5-3B-Instruct 用 LLaMAFactory 做 LoRA SFT 和权重合并，最后再进入 vLLM 服务部署和吞吐优化。

我之所以采用这种做法，而不是把项目只讲成“我调了几个 vLLM 参数”，主要有三个原因。第一，这个项目本身确实包含数据生成、训练格式对齐、LoRA 微调和推理部署，不只是后半段服务调优；第二，训练和推理模板保持一致，本身也会影响后面的 prefix caching 和整体效果；第三，从面试角度看，这样更符合项目原貌：训推都做了，只是最后的优化目标落在吞吐上。

从实现上看，这个项目可以分成两部分。前半部分是训练侧：我做了教材 PDF 到双语 QA 的数据构造、去重、ShareGPT(messages) 格式整理，并基于 Qwen2.5-3B-Instruct 用 LLaMAFactory 做 LoRA 微调和权重合并。后半部分是推理侧：我用 FastAPI 提供 `/predict` 接口，底层推理引擎用 vLLM，围绕 continuous batching、prefix caching、确定性解码，以及 `max_model_len`、`max_num_seqs`、`gpu_memory_utilization` 这些参数做实验，再用离线评测脚本对比不同配置下的延迟和 token/s 变化。它的实际意义不在于“我做了一个特别大的系统”，而在于我把训练和推理这两段都真正跑了一遍，而且知道最后的优化重点落在哪。

如果写进简历，我一般会压成下面这两种说法：

- 短版一句话：基于教材 PDF 构建 GPU 领域双语问答数据，使用 LLaMAFactory 对 Qwen2.5-3B-Instruct 进行 LoRA 微调与权重合并，并基于 FastAPI + vLLM 完成离线推理部署与吞吐优化。
- 稍长一点：围绕 GPU 领域问答场景，完成 ShareGPT(messages) 数据整理、Qwen2.5-3B 的 LoRA SFT、合并权重导出，以及基于 vLLM 的 continuous batching、prefix caching 和确定性解码优化，最终面向离线推理吞吐做实验与评测。

## 先准备这 10 个关键问题

如果时间有限，建议先把下面这 10 个问题讲顺。它们基本覆盖了这个项目最容易被问到、也最值得主动讲的点。

### 1. 你这个项目到底是做什么的？

这是一个 GPU 领域双语问答模型训练与 vLLM 推理优化项目。我先做了教材 PDF 到双语 QA 数据的整理，再基于 Qwen2.5-3B 用 LLaMAFactory 做 LoRA 微调和权重合并，最后用 FastAPI + vLLM 做离线推理部署和吞吐优化。

### 2. 这个项目你最想突出哪一部分？

我最想突出的是它其实是一个训推都做了的项目，只不过最后的目标落在推理吞吐上。所以我会先把数据构造、LoRA 微调和权重合并讲清楚，再把重点放到 vLLM 的 continuous batching、prefix caching、批量请求、解码策略和参数权衡上。

### 3. 为什么不是只讲“我用了 vLLM”，还要做数据和 LoRA？

因为如果没有一个基本可用的模型和场景，后面的推理实验就会比较空。这个项目里，数据构造和 LoRA 微调不是可有可无的铺垫，而是整个实验链路的一部分。只是相对来说，最终评测目标更关注推理吞吐，所以我会在面试里把推理侧展开得更细一些。

### 4. 你为什么选择 vLLM，而不是直接用 transformers？

因为这个项目本来就是围绕吞吐做优化，vLLM 在这类场景下更适合做实验。特别是 continuous batching 和 prefix caching 这些机制，比较适合用来观察不同配置对 token/s 的影响。

### 5. continuous batching 的价值是什么？

它的核心价值是提高 GPU 利用率。静态 batch 容易被最长序列拖住，而 continuous batching 可以更灵活地把不同请求拼到一起，让 GPU 少一点空转。

### 6. prefix caching 在你这里为什么有效？

因为我这边很多请求都会复用比较固定的 system prompt 和模板，所以前缀相似度比较高。这样更容易命中 prefix caching，减少重复 prefill，吞吐上通常会更有优势。

### 7. 你为什么用了确定性解码？

因为这个项目更偏离线评测和推理速度实验，我更看重稳定和可复现，而不是输出多样性。像 `temperature=0.0`、`top_k=1` 这种设置，能让不同配置之间更容易做对比。

### 8. 你怎么证明这不是一个套壳项目？

我不会说自己做了什么特别大的系统，但它也不是简单包一层接口。因为我确实自己做了数据整理、LoRA 微调、LLaMAFactory 训练、权重合并、FastAPI 服务、vLLM 参数实验和离线评测，把训练到推理这条链路真正动手跑了一遍。

### 9. 这个项目最容易被质疑的点是什么？

最容易被质疑的是指标口径，比如准确率怎么定义、token/s 是在什么硬件和并发条件下测的。另一个点是训练部分虽然做了，但如果我讲得太轻，会显得和原项目不一致；如果讲得太重，又会像在夸大训练深度。所以我会把边界讲清楚：训推都做了，但最终优化目标是推理吞吐。

### 10. 如果上生产，你最先会改什么？

我会优先补两类东西。第一类是更系统的服务压测和监控，比如更清晰的 P50/P95 延迟、不同并发下的表现；第二类是更扎实的评测和对比实验，把测试条件和收益讲得更严谨一些。

这份文档按三类岗位来整理：

1. 大模型开发
2. 大模型应用
3. Agent 应用

每个问题都尽量贴合这个项目本身，回答长度控制在面试中比较自然的范围内，避免太空泛，也避免讲得太满。

## 一、大模型开发方向

### 1. 你这个项目里，大模型开发主要体现在哪些环节？

主要体现在三块。第一块是领域数据构造，包括知识点抽取、问答生成、去重和训练格式对齐；第二块是基于 Qwen2.5-3B 的 LoRA SFT 微调；第三块是围绕模型部署后的效果和吞吐做参数权衡。也就是说，这个项目不是只做了推理调参，训练这部分我也确实做了，只是最终优化目标落在推理吞吐上。

### 2. 为什么你选择 Qwen2.5-3B-Instruct？

因为这个项目既要考虑一定的领域效果，又要考虑部署和评测时的加载成本、显存占用和推理吞吐。7B 在这类环境里更容易带来加载和推理压力，0.5B 级模型又不太容易把效果拉起来，所以 3B 对我来说是效果和部署成本之间比较合适的折中。

### 3. 为什么你没有直接做全参数微调，而是选 LoRA？

因为这个项目的目标不是做最大规模训练，而是做一个训练和部署都能跑通、并且适合离线评测环境的方案。LoRA 的好处是训练成本低、显存更友好、迭代更快，同时对这种领域增强任务通常已经够用。实现上我用的是 LLaMAFactory 这类高效微调框架去配置和执行 LoRA SFT，而不是自己从零重写训练框架，这样更符合课程项目和工程验证的实际需求。

### 4. 你的数据为什么要统一成 ShareGPT(messages) 格式？

因为我的推理侧最终是按 chat template 去组织 system、user、assistant 结构的，如果训练数据直接对齐成 messages 格式，就能尽量减少训练分布和推理分布之间的偏移。换句话说，我希望模型在训练时看到的输入组织方式，和部署后实际看到的尽可能一致。

### 5. 你这个数据集构造过程里，最关键的设计是什么？

我觉得最关键的是“先知识点、后问答”的两阶段生成方式。因为如果直接让模型从整章内容里大批量生成 QA，很容易出现覆盖不均、重复严重、重要概念缺失的问题。先抽知识点，相当于先做一层概念地图，再围绕每个点生成问答，数据结构会更可控。

### 6. 你是怎么降低重复样本问题的？

我做了几层处理。先对 instruction 做规范化，比如 NFKC、小写化、空白折叠；然后再结合相似度阈值做过滤。如果某个知识点名称已经被现有 instruction 充分覆盖，我会把预算留给别的概念，而不是继续堆相似问题。

### 7. 你怎么理解这个项目里的“数据质量”问题？

我觉得数据质量不只是“看起来像样本”，而是至少要满足三点：第一，问题本身能脱离上下文独立理解；第二，覆盖的概念足够全面，不只盯着几个高频术语；第三，问答之间不要高度重复。因为对小模型做领域增强时，低质量合成数据很容易把训练变成记模板，而不是学知识结构。

### 8. 这个项目里你最担心的数据风险是什么？

我最担心的是两类风险。第一类是合成数据里可能带入生成模型自己的偏差，第二类是训练集和评测集之间如果划分不严，容易让结果显得比真实情况更乐观。所以如果后续继续做，我会把数据划分、人工抽检和错误分析做得更严格。

### 9. 你的训练结果怎么判断是有效的？

我不会只看 loss 降了没有，而是会结合离线评测结果、具体 bad case 和推理阶段的实际表现一起看。因为 loss 下降只能说明模型拟合了训练分布，但不能直接说明部署后真的回答更好。所以我更看重训练和推理是否形成了闭环验证。

### 10. 如果让你继续深化成更强的大模型开发项目，你会先补什么？

我会优先补三件事：第一，做更严格的数据划分和 baseline 对比；第二，补充人工抽检和错误分析；第三，把准确率指标的定义、评测流程和 ablation 做完整。这样项目就不只是“做出来了”，而是更像一个可解释的训练实验。

## 二、大模型应用方向

### 11. 这个项目最核心的应用价值是什么？

最核心的价值是：它不是只停留在离线训练，也不是只停留在单纯部署，而是把数据构造、LoRA 微调、权重合并和推理优化串了起来。它不算特别完整的大模型应用系统，但作为一个训推都做过、并且以推理吞吐为目标的项目，边界是比较清楚的。

### 12. 你为什么要自己做 FastAPI 服务，而不是只离线跑脚本？

因为脚本只能证明模型能生成，但服务接口才能证明它有明确输入输出契约，也方便做后续压测和批量请求实验。特别是这个项目的重点之一就是吞吐优化，所以我需要一个统一的调用入口，把模型能力和服务性能真正连起来。

### 13. `/predict` 同时支持单条和批量输入，这个设计有什么意义？

因为单条请求更接近日常调用方式，批量请求更接近吞吐优化场景。如果我只支持单条接口，就很难把多个请求合并成一次 vLLM 调用，也很难观察 batch size 对 token/s 和端到端延迟的影响。所以这个接口设计其实是为后面的调度优化服务的。

### 14. 你为什么选择 vLLM，而不是直接用 transformers 起服务？

因为这个项目的评测目标非常明确，就是要追求更高的推理吞吐。vLLM 在这种场景下的优势比较明显，尤其是 continuous batching 和 prefix caching 这些机制，更适合做高并发、高吞吐的生成服务。相比之下，直接用 transformers 更容易跑通，但在吞吐和调度能力上不够有优势。

### 15. 你项目里 continuous batching 的价值是什么？

它的价值在于提高 GPU 利用率。静态 batch 容易被最长序列拖住，而 continuous batching 可以把不同请求在 token 级别更灵活地调度，让新的请求及时补进来，减少空转时间。对这种要追求 token/s 的场景来说，这是非常关键的优化点。

### 16. prefix caching 在你这里为什么有效？

因为我的训练和推理都复用了比较固定的 system prompt 和 chat template，这就意味着很多请求在前缀部分是相同的。只要前缀一致，就有机会复用之前已经计算过的 prefill 结果，从而减少重复计算。对这种领域问答场景来说，这个前提是比较容易满足的。

### 17. 你为什么要强调训练和推理的模板一致？

一方面是为了减少训练分布和推理分布不一致带来的偏差，另一方面也是为了让固定前缀更容易复用，从而提高 prefix caching 的收益。也就是说，这个设计既有模型效果层面的考虑，也有系统性能层面的考虑。

### 18. 你为什么选确定性解码？

主要有两个原因。第一，离线评测更需要可复现和稳定，而不是多样化输出；第二，确定性解码下输出方差更小，服务表现更稳定，也更方便我比较不同配置的真实差异。所以这个任务里我更看重稳定性，而不是创意性。

### 19. 你的吞吐优化是怎么做权衡的？

我不是单纯把所有并发参数拉满，而是围绕显存约束做平衡。比如 `gpu_memory_utilization` 会影响 KV cache 可用空间，`max_model_len` 影响单请求的上下文上限，`max_num_seqs` 影响可同时驻留的序列数。这几个参数本质上是在“单条能力”和“整体吞吐”之间做权衡。

### 20. 你为什么要做预热？

因为首请求通常会受到模型初始化、缓存建立和执行路径冷启动的影响，导致时延波动比较大。如果评测一开始就直接压测，不一定能反映服务稳定阶段的真实表现。所以我会先做一次轻量预热，让缓存和关键状态先建立起来。

### 21. 这个项目最能体现你哪些应用工程能力？

我觉得主要是四点：第一，能把训练后的模型做成接口服务；第二，知道怎么围绕 vLLM 做吞吐优化；第三，能把评测脚本和服务联动起来；第四，能清楚解释为什么要做这些工程取舍，而不是只会把模型跑起来。

### 22. 如果放到真实业务里，这个项目最先要补什么？

我会优先补三块：第一，更严格的评测和监控体系；第二，更清晰的服务压测条件和延迟指标，比如 P50/P95；第三，更真实的业务查询和 bad case 闭环。因为课程项目更偏验证链路，走向业务还需要补“长期稳定性”和“真实场景可信度”。

## 三、Agent 应用方向

### 23. 这个项目能不能算 Agent 项目？

严格来说，我不会把它当成 Agent 项目主打。因为它没有明显的工具调用闭环、任务规划、记忆、反思或者多步骤自主执行机制。它更像一个领域模型训练和高吞吐推理优化项目。如果去面 Agent 岗，我会把它放在“底层模型和服务能力”这个位置，而不是强行说它是 Agent 系统。

### 24. 那它和 Agent 岗完全没关系吗？

也不是完全没关系。它至少能证明我对模型服务、领域增强、接口设计和性能优化这些底层能力是有实践的。而 Agent 系统一旦真的落地，最终也离不开模型服务、推理成本和调用稳定性。所以它可以作为 Agent 项目的基础能力证明，但不能当作完整 Agent 项目本身。

### 25. 如果面试官问“你这个项目为什么不算 Agent”，你怎么答？

我会比较坦诚地说：因为它没有把重点放在任务规划、工具调用和多轮自主执行上，而是放在了领域数据、微调和推理部署优化上。我觉得面试里主动把边界讲清楚，比硬包装成 Agent 更有说服力。

### 26. 如果要把这个项目往 Agent 方向升级，你会怎么做？

我会先在问答系统上加“检索工具调用 + 查询改写 + 结果审查”这类最小闭环，再进一步做多步任务拆解和执行记录。比如用户问一个复杂 GPU 性能问题时，系统先判断是否需要查教材、是否需要查外部资料，再组织回答和证据。这种升级路径会更自然，而不是完全推倒重来。

### 27. 这个项目对 Agent 岗最大的短板是什么？

最大的短板是没有真正的 agent loop，也没有围绕工具调用质量、规划成功率、任务完成率去做评测。也就是说，它有模型能力和服务能力，但还没有把这些能力组织成一个可自主推进任务的系统。

## 四、跨岗位高频追问

### 28. 你这个项目最容易被质疑的点是什么？

我觉得主要有三个。第一，准确率指标的定义如果讲不清，很容易被质疑；第二，tokens/s 如果不绑定硬件、并发和上下文条件，也容易显得不够严谨；第三，合成数据比例比较高，面试官可能会追问数据质量和泛化能力。所以这些点我会主动讲边界，而不是回避。

### 29. 面试官如果追问 accuracy=0.4179 是什么，你怎么答？

我会先把这个指标的评测口径讲清楚，再强调它的意义不是“模型已经很强”，而是用于比较不同训练和部署方案的相对效果。如果口径不讲清楚，单独报这个数字没有说服力；所以我会重点解释数据来源、评测方法和对比对象。

### 30. 面试官如果问 16623.62 tokens/s 高不高，你怎么答？

我不会直接说高或低，而是先说这个数字必须结合具体测试环境理解，比如 GPU 型号、batch、大概输入长度、是否离线批量、prefill 和 decode 的占比。这个指标对我来说最重要的价值不是绝对数值，而是证明我知道怎么去做系统级吞吐优化和实验对比。

### 31. 你文档里说动态 LoRA 更慢，这个点怎么讲才稳妥？

更稳妥的说法不是“动态 LoRA 通常会慢 7 到 8 倍”，而是“在我这次项目的配置和负载下，动态挂载适配器带来了明显的额外开销，所以最终为了吞吐最大化，我选择了合并权重后的单模型部署方案”。这样更符合工程表述，也不容易被抓住绝对化说法追着问。

### 32. 你怎么证明这不是一个“套壳项目”？

我会从四个角度证明。第一，我自己设计并实现了数据构造流程和去重策略；第二，我做了 LoRA 微调和权重合并，而不是只调 API；第三，我把模型真正部署成了服务，并围绕 vLLM 做了吞吐优化；第四，我还有离线评测脚本来验证不同配置的效果。把这些连起来，它已经不是简单包一层接口了。

### 33. 这个项目最强的卖点是什么？

我觉得最强的卖点是它既不是只训练，也不是只部署，而是把数据构造、LLaMAFactory + LoRA 微调、权重合并、vLLM 服务和离线评测串成了一条完整实验链路。再往里讲的时候，我最能展开的还是 vLLM 这一侧的速度优化，因为这是项目最终的核心目标。

### 34. 这个项目最薄弱的点是什么？

最薄弱的点是训练部分虽然做了，但深度还不算特别强，比如更系统的 baseline、消融和错误分析还可以继续补；另外评测体系也还不算特别完善。所以面试里最好把边界说清楚：这个项目的优势是训推链路完整、目标明确，不是训练研究本身做得特别深。

### 35. 如果让你只用 1 分钟介绍这个项目，你会怎么说？

我会说：这是一个 GPU 领域双语问答模型训练与 vLLM 推理优化项目。我先基于教材 PDF 整理问答数据，统一成 ShareGPT(messages) 格式，再基于 Qwen2.5-3B 用 LLaMAFactory 做 LoRA 微调和权重合并。然后我用 FastAPI + vLLM 搭了离线推理服务，重点测试 batch、prefix cache 和解码策略对吞吐的影响。

### 36. 如果你重新做一次，你最想补哪一块？

我最想补的是更扎实的评测和对比实验。比如把不同 vLLM 配置的对比做得更系统一些，再把测试条件写得更清楚。因为对这种项目来说，讲清楚“怎么测、为什么快了”比把项目包装得很大更重要。

### 37. 这个项目最适合拿去面什么岗？

我觉得最适合的是大模型应用开发、模型服务部署、推理优化这几类岗位；如果是偏训练工程的岗位，它也能提供一定支撑，因为数据构造、LoRA 微调和权重合并这部分我确实做了；如果是纯 Agent 应用岗，最好再配一个更相关的项目一起讲。

### 38. 这个项目最能说明你和别人有什么区别？

我觉得区别不在于我用了什么特别新的东西，而在于我没有只停留在单点实验，而是把数据构造、LoRA 微调、权重合并、服务搭建、参数调优和离线评测这几步都自己跑过一遍。这个项目不算特别大，但比较完整，也比较实在。

## 五、建议你怎么使用这份文档

### 方法 1：先背“详细版介绍”，再压缩成“简历版”

建议你先把上面的详细版项目介绍讲顺，因为它能帮你建立完整故事线。等你能顺畅讲 3 分钟版本后，再压缩成 30 秒和 1 分钟版本，面试时会自然很多。

### 方法 2：按岗位有侧重点地讲

- 面试大模型开发：重点讲数据构造、ShareGPT 格式、LoRA、模型选型、数据质量
- 面试大模型应用：重点讲 FastAPI、vLLM、continuous batching、prefix caching、压测和部署
- 面试 Agent 应用：坦诚说明这不是 Agent 主项目，重点把它讲成 Agent 底层能力支撑

### 方法 3：重点准备这 6 个问题

优先熟悉这些：

1. 为什么选择 Qwen2.5-3B、LoRA 和 LLaMAFactory
2. 为什么要统一成 ShareGPT(messages)
3. 为什么要合并 LoRA 而不是动态挂载
4. continuous batching 和 prefix caching 分别解决什么问题
5. tokens/s 和 accuracy 具体怎么理解
6. 这个项目为什么不算 Agent，但为什么依然有岗位价值

### 方法 4：简历上不要写得太满

建议你在简历里重点写：

- 做了什么
- 用了什么技术
- 解决了什么问题
- 指标是在什么背景下得到的

不要一上来就写特别绝对的话，比如：

- “动态 LoRA 一定更慢”
- “prefix cache 一定大幅提升吞吐”
- “模型准确率显著提升”

更稳妥的写法应该是“在本项目配置和工作负载下观察到……”。这样更像工程师，也更经得起追问。

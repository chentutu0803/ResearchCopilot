# 面试提问与回答（按面试官视角整理）

## 项目简介

我的项目叫 ResearchCopilot，中文可以叫研究副驾驶。它本质上是一个面向研究类任务的深度研究助手。我大概用一周左右把第一版完整做出来，重点不是做一个看起来很花哨的问答工具，而是把“拆题、检索、写作、审查、追踪”这一整套流程真正跑通。

它主要想解决一个很实际的问题：普通大模型虽然能回答问题，但一到研究类任务，经常会出现结果不稳定、来源说不清、过程没法解释的问题。为了解决这个问题，我把系统拆成了 `Planner -> Researcher -> Writer -> Reviewer` 这条主链路。Planner 负责把主题拆成 3 到 5 个子问题，Researcher 负责调用 Web Search 和本地知识库做检索，Writer 基于证据生成带引用的结构化报告，Reviewer 则先抽取引用、覆盖率、证据使用情况和来源质量这些结构化特征，再把它们和正文一起交给 LLM 做一次统一审查，最后输出报告、参考资料、Trace 和运行工件。

我之所以采用这种方法，而不是直接让模型一次性回答，主要有三个原因。第一，研究任务天然是多阶段问题，先拆题、再找证据、再写作，会比端到端生成更稳定；第二，加入 Hybrid Retrieval、Query Rewrite、Reviewer 和 Trace 后，系统更容易调试，也更方便做评测；第三，这样做更能体现大模型开发、RAG 工程、Agent 工作流和系统化调试能力，而不只是简单调 API。

从实现上看，这个项目用了 Hybrid Retrieval，也就是 BM25、Dense Retrieval 和 Re-rank 这一套组合，来提高检索质量；用了 Citation Checker 和统一 Reviewer 审查流来提高输出可信度；用了 Trace、运行工件和评测脚本来提高系统可解释性。它的实际意义在于：不仅能生成研究报告，还能回答“这个结论来自哪里、为什么这么写、这个系统到底是怎么工作的”。

## 先准备这 10 个关键问题

如果时间有限，建议先把下面这 10 个问题讲顺。它们基本覆盖了这个项目最容易被问到、也最值得主动讲的点。

### 1. 你这个项目到底是做什么的？

我的项目叫 ResearchCopilot，它是一个面向研究类任务的深度研究助手。用户输入一个主题后，系统会先拆成几个子问题，再去搜索和检索资料，整理证据之后生成带引用的报告，最后再做审查和反馈。它和普通问答系统最大的区别是，不只是给一个结果，而是尽量把结果和来源、过程、评测一起做出来。

### 2. 为什么不用 LangChain / Dify，为什么自己写？

因为这个项目的核心规模其实不大，我更想把工作流、工具调用、结构化输出、Trace 和质量控制这些关键部分自己实现出来。这样一方面我对系统内部机制更清楚，另一方面也更容易解释为什么这样设计。如果一开始就上重框架，很多关键细节会被抽象掉。

### 3. 为什么不用完全自治多智能体，而是 workflow？

因为这个任务本质上是线性依赖的：先规划，再检索，再写作，再审查。对这种任务来说，workflow 更可控，也更容易调试和评测。如果一开始就做成完全自治多智能体，复杂度会上升很多，但收益不一定匹配。我的选择是先把闭环做实，再考虑以后把某些阶段并行化。

### 4. 为什么要做 Hybrid Retrieval，而不是只做向量检索？

因为纯向量检索更擅长语义相似，但对专有名词、缩写词、版本号这类精确匹配场景不够稳。BM25 刚好能补这部分短板，所以我用了 BM25 + Dense Retrieval 的混合召回，再加一层 Re-rank。这样做的目标是同时兼顾召回覆盖率和前几条结果的相关性。

### 5. Re-rank 的价值是什么？

混合召回之后，候选结果通常还是比较杂，不能直接拿去写报告。Re-rank 的作用是把最相关的证据尽量排到前面，让 Writer 用到的上下文更干净。简单说，召回负责“别漏”，精排负责“前面几条要准”。

### 6. 你怎么降低幻觉，怎么让结果更可信？

我主要做了三层约束。第一层是检索增强，先找证据再写；第二层是写作时要求引用编号，把关键结论和证据关联起来；第三层是 Reviewer 和 Citation Checker 再检查引用是否合法。也就是说，我不是只靠 prompt 去压模型，而是把证据约束做进了系统流程里。

### 7. Reviewer 的实际价值是什么？

Reviewer 不只是“再看一遍结果”，它是整个闭环里的质量控制点。它会检查引用覆盖、结构完整性和基本质量，如果结果不达标，系统不会只在原稿上修几句话，而是会带着反馈回到前面的规划、检索和写作阶段再走一轮。这样 Reviewer 才真正能影响上游，而不是停留在表层修补。

### 8. 你怎么避免 Agent 系统无限循环？

我在几个关键位置都加了边界。Researcher 有最大搜索轮数，Reviewer 失败后回到前面的闭环也有最大次数，整个 Workflow 还有清晰状态流转。所以这个系统不是“想搜多久搜多久”，而是会在可控范围内重试，达到上限后强制停止并保留结果状态。

### 9. 你怎么证明这不是一个套壳项目？

我会从三个角度回答。第一，它不是单次调用模型，而是完整工作流；第二，它有工具调用、失败恢复、引用约束和 Reviewer 闭环；第三，它有 Trace、运行工件、检索评测和工作流消融。也就是说，我做的不只是页面和 API 对接，而是把一个研究型任务做成了可以运行、可以验证、可以复盘的系统。

### 10. 如果上生产，你最先会改什么？

我会优先改三件事。第一是缓存和并发，减少重复检索带来的延迟和成本；第二是更稳定的搜索服务和更完善的错误处理；第三是更大规模、更细粒度的评测体系，比如更严格的 claim-level 一致性检查。因为从原型走向业务，最先暴露的问题通常不是功能不够，而是稳定性、成本和评测深度不够。

这份文档按三类岗位来整理：

1. 大模型开发
2. 大模型应用
3. Agent 应用

每个问题都尽量贴合 `ResearchCopilot` 这个项目本身，回答长度控制在面试中比较自然的范围内，避免太空泛，也避免太绕。

## 一、大模型开发方向

### 1. 你这个项目里，大模型主要负责哪些环节？

大模型主要负责四件事：主题拆解、检索过程中的语义辅助、报告写作，以及 Reviewer 里的统一审查判断。当前版本的 Reviewer 不是纯 LLM judge，也不是规则先审一次再让 LLM 复读一遍，而是先抽取结构化规则特征，再让 LLM 结合正文在同一次 call 里给出最终判断。这样既保留了质量控制的可解释性，也补上了纯规则对细粒度语义问题不够敏感的短板。

### 2. 为什么你没有把这个项目做成“用户问题直接问模型”的简单问答系统？

因为研究型任务和普通问答不一样，它通常需要多步检索、证据整理和结构化输出。如果直接让模型一次性回答，最大问题是结果不稳定，而且很难追溯来源。我这个项目的目标不是做“能回答”，而是做“回答有证据、过程可解释”。

### 3. 你为什么选择 Hybrid Retrieval，而不是只用向量检索？

因为纯向量检索对语义相似表达比较友好，但对专有名词、缩写词、版本号、关键词这种精确匹配场景不够稳。BM25 刚好能补这部分短板，所以我用了 BM25 + Dense Retrieval 的混合召回，再加一层精排。这个设计更接近实际 RAG 系统的工程做法。  
如果面试官继续追问中文分词或 tokenizer，我会补一句：`Qdrant` 官方现在已经支持更标准的 `BM25` 文本检索和 `multilingual tokenizer`，做中文/多语言会更自然，所以我现在会优先走这条路，而不是自己在本地手写一个简单 tokenizer。

### 4. 你项目里的 Dense Retrieval 是怎么做的？

我这里不是普通 RAG，而是 `BM25 + Qdrant dense retrieval + rerank` 的混合检索。Dense 这层我会把 chunk 做 embedding 后存进 `Qdrant`，主要负责语义召回；关键词和专有名词这部分交给 `BM25`，最后再做一层精排。所以 dense retrieval 不是单独工作，而是和 `BM25` 一起服务于整个 hybrid retrieval。

### 4.1 你的本地知识库是怎么建的？文档从哪里来？

本地知识库是我离线构建的，不是在线临时抓。我会先把 `data/raw` 下面的 `md/txt` 文档统一读进来，按段落切 chunk，默认 `chunk size` 大概是 `420`，`overlap` 是 `60`，然后把每个 chunk 变成一个带 `title`、`snippet`、`content` 和原文件路径信息的 `EvidenceItem`。文档来源一部分是我自己整理的主题资料，另一部分是从 `LangGraph`、`Qdrant`、`Milvus`、`LlamaIndex`、`vLLM`、`OpenAI Cookbook` 这些相关官方仓库和文档里同步下来的，核心思路就是围绕项目涉及的 agent workflow、RAG、retrieval、vector DB、embedding 和推理框架去找高质量、相对稳定的资料。

### 5. 为什么还要做 Re-rank？混合召回之后不够吗？

混合召回解决的是“不要漏”，但召回出来的候选结果通常还是比较杂。Re-rank 的作用是把最相关的结果排到前面，提升最终输入到写作阶段的证据质量。简单说，召回更关注覆盖率，精排更关注前几条结果的准确性。

### 6. 你的评测为什么选 Hit@3 和 MRR@5？

因为这个项目目前是一个面试展示型工程项目，不是大型 benchmark 系统，所以我优先选了容易解释、和检索排序质量强相关的指标。Hit@3 能看前几条里有没有命中，MRR@5 能看正确结果排得靠不靠前。这两个指标足够支撑我说明检索策略的价值。

### 7. Reviewer 在你的项目里解决了什么问题？

Reviewer 主要解决两个问题：第一，检查报告是否真的有足够引用支撑；第二，检查每个子问题是不是被充分回答。当前版本里它会先抽取结构化特征，比如无效引用、子问题覆盖、每个子问题实际用了几条证据、每个子问题对应来源的 source map 和质量标签，然后把这些特征和正文一起交给 LLM 做一次统一判断。如果它判定不通过，系统不会只在原稿后面补几句，而是会带着反馈回到前面的规划、检索和写作阶段再走一轮。

### 8. 如果模型输出不稳定，尤其是 JSON 结构化输出不稳定，你怎么处理？

我做了两层处理。第一层是提示词约束，明确要求返回固定 JSON 结构；第二层是解析和异常处理，如果解析失败就降级到模板逻辑，而不是让整个流程中断。这样即使模型波动，系统也不会直接崩。

### 9. 你怎么降低幻觉问题？

我主要从三个层面控制。第一是检索增强，让模型先基于证据再写；第二是写作时要求引用编号；第三是 Reviewer 和 Citation Checker 再检查引用是否合法。也就是说，我不是只靠 prompt 压模型，而是把“证据约束”做进了系统流程里。

### 10. 如果让你继续往“大模型开发”方向深化这个项目，你会优先做什么？

我会优先做两件事。第一是换成更真实的本地 embedding 和 reranker，并扩大评测集，让检索实验更有说服力；第二是把 claim 和 evidence 的映射做得更严格，进一步提高引用约束能力。这样会更贴近真正的大模型系统优化工作。

### 11. 你现在的 embedding 和 reranker 是完全真实模型吗？

当前项目是双路径设计。默认模式优先保证轻量可跑，所以有轻量 fallback；如果安装增强依赖，就可以切到更真实的 embedding 和 reranker 路径。这样做不是为了回避问题，而是为了兼顾可运行性和后续可扩展性。

### 12. 为什么你的评测结果看起来都比较高，会不会太理想化？

这是因为当前知识库规模比较小、领域比较集中，所以评测更像一个“验证方法有效性”的小型实验，而不是大规模 benchmark。我在面试里会主动说明这一点：这些指标能证明设计思路成立，但还不能等同于生产环境下的完整结论。

### 13. 你怎么理解“评测有效”和“系统真实可用”之间的区别？

评测有效说明某个模块设计在当前数据和指标下是有价值的，比如检索排序更好了；但系统真实可用还要看稳定性、错误处理、成本、延迟和数据规模。我的项目现在已经覆盖了评测和可运行原型，但如果走向业务，还需要更大规模的数据和更严格的线上验证。

## 二、大模型应用方向

### 14. 这个项目最核心的产品价值是什么？

最核心的价值是：它不是只给一个答案，而是给一个有证据、可追溯、可复盘的研究结果。对很多大模型应用来说，用户最担心的不是“模型会不会说”，而是“它说得靠不靠谱”。这个项目就是围绕“可信输出”来设计的。

### 15. 你为什么选择 Streamlit，而不是做成前后端分离？

因为这个项目的目标是尽快把完整链路做出来并展示清楚。Streamlit 很适合这种场景：开发成本低、演示效果直观、能快速把输入、过程、输出和 Trace 放在同一个页面里。对当前这个项目来说，这个选择更务实。

### 16. 你项目里的用户体验设计重点是什么？

我更关注“让面试官一眼看懂系统在干什么”。所以页面里我重点展示了输入主题、子问题拆解、证据列表、最终报告、Reviewer 结果和 Trace 日志。这样用户不需要看代码，也能理解这个系统不是黑盒，而是一个分阶段工作的研究助手。

### 17. Web Search 和本地知识库为什么要同时保留？

因为它们解决的是两个不同问题。Web Search 更适合补充最新信息和开放域资料，本地知识库更适合提供稳定、可控、可复现的高质量证据。把两者放在一起，一方面能增强系统能力，另一方面也方便我在面试时说明“开放搜索”和“本地 RAG”的区别。

### 18. Query Rewrite 是怎么触发的？

在我的设计里，它不是无条件触发的，而是在检索结果为空或者证据明显不足的时候才继续改写 query。并且整个过程有上限，不会无限改。这样做的目标是增强鲁棒性，而不是让系统无休止搜索。

### 19. 你的项目怎么保证失败时还能继续工作？

我做了比较明确的 fallback 设计。比如 Web Search 失败时不会让整个流程挂掉，知识库仍然可以提供结果；如果没有 API Key，也可以走启发式模式把流程完整跑通。也就是说，我希望这个系统在资源不完整的情况下依然能工作，而不是只在理想环境下演示成功。

### 20. 为什么你认为 Trace 很重要？

因为大模型应用里最麻烦的问题通常不是“没有结果”，而是“结果为什么不对”。Trace 能帮助我定位到底是主题拆解不对、检索不够、写作约束失效，还是 Reviewer 没触发。它相当于是这个系统的调试基础设施。

### 21. 你为什么把运行工件保存成 JSON？

因为 JSON 对调试和复盘都很友好。它可以完整保存主题、子问题、证据、最终报告和 Trace，后续无论是做问题定位、结果对比，还是给面试官展示系统中间态，都很方便。相比只保存一篇最终文章，这种结构化工件价值更高。

### 22. 这个项目如果放到真实业务里，最先要补什么？

我觉得最先要补三件事。第一是缓存和并发，避免重复搜索浪费时间；第二是更稳定的搜索服务和更完整的错误处理；第三是更严格的评测体系，尤其是 claim-level 的事实一致性检查。这样才能从当前原型进一步走向业务应用。

### 23. 这个项目和普通“RAG 问答”最大的区别是什么？

最大的区别是它不是只回答一个问题，而是把研究任务拆成一个多步骤流程。它会先做问题拆解，再检索，再整理证据，再写报告，再做审查。所以它更像一个研究型工作流系统，而不是普通的“检索后直接回答”。

### 24. 你的页面为什么还要展示中间过程，不怕太复杂吗？

如果是面向普通用户的成熟产品，页面可以更简洁；但这个项目首先是一个研究型 Agent 系统原型，所以展示中间过程反而是价值的一部分。它能让面试官或使用者看到系统怎么拆题、怎么检索、怎么形成最终结果，这对建立信任很重要。

### 25. 你怎么平衡“结果质量”和“响应速度”？

我的做法是给每个可能变慢的环节加边界，比如搜索轮数有限、Reviewer 轮数有限、检索 top-k 有限制。也就是说，我不会一味追求结果更长更复杂，而是通过上限控制把系统维持在一个可演示、可接受的响应范围内。

### 26. 这个项目最适合什么类型的应用场景？

我觉得它最适合需要“结构化研究输出”的场景，比如技术综述、竞品调研、行业资料汇总、内部知识整理。这类任务的共同点是：用户不只关心最终结论，还关心来源、逻辑结构和可复查性。

## 三、Agent 应用方向

### 27. 为什么你选择 workflow，而不是完全自治的多智能体架构？

因为这个任务本质上是强线性依赖的：先规划，再研究，再写作，再审查。对这种任务来说，workflow 更容易控制，也更容易调试。如果一开始就上完全自治的多智能体，复杂度会明显提高，但收益不一定匹配，尤其是对当前这个项目规模来说并不划算。

### 28. 那你为什么还要保留 Planner、Researcher、Writer、Reviewer 这种 Agent 划分？

因为即使我用的是 workflow，我仍然希望每个阶段有清晰职责。Planner 负责拆题，Researcher 负责找证据，Writer 负责写作，Reviewer 负责质量控制。这样模块边界清晰，后续扩展也更方便，比如以后可以把 Researcher 并行化。

### 29. Researcher 在你这里算不算真正的 Agent？

我认为它算一个“受限 Agent”，而不是完全自治 Agent。它有明确目标、会调用工具、会根据结果调整 query，也有停止条件。但它的边界是由 workflow 控制的，所以它更偏工程可控，而不是自由探索式智能体。

### 30. 你的 Agent 系统怎么避免无限循环？

我用了非常明确的边界控制。Researcher 有最大搜索轮数，Reviewer 失败后触发的重新规划闭环也有上限，Workflow 本身还有清晰状态流转。也就是说，每个“会循环”的模块都有上限，这样系统天然可终止，不会陷入失控状态。

### 31. Reviewer 为什么有必要单独做成一个模块？

因为写作和评审其实是两种不同任务。Writer 更关注组织和表达，Reviewer 更关注事实支撑、引用覆盖和结构完整性。把它们分开后，职责更清晰，也更容易解释为什么需要一个额外的质量控制环节。

### 32. 你怎么评价自己这个项目的“Agent 味道”够不够？

如果按最严格定义，它不是那种完全自治、多智能体协商式的系统；但如果从工程角度看，它已经具备了 Agent 应用的关键特征：任务分解、工具调用、状态流转、失败恢复、质量评审和可观测性。所以我会说它是一个偏工程化、可控型的 Agent Workflow 项目。

### 33. 如果面试官说“你这不就是工作流，不算 Agent”，你怎么答？

我会先认可这个问题的一部分：对，我没有刻意把它包装成完全自治多智能体。因为我的目标不是炫架构，而是做一个稳定、可解释、可评测的研究型系统。但它依然具备 Agent 应用的关键能力，比如工具调用、动态检索、质量评审和任务阶段协作，所以它是一个更偏工程落地的 Agent 方案。

### 34. 你项目里的 Agent 通信是怎么做的？

严格说我这里没有做复杂的自然语言消息总线，而是通过统一的数据结构在各阶段之间传递状态，比如子问题、证据项、报告草稿和审查结果。这样做更简单，也更适合这个项目规模。对面试来说，这也能体现我对“什么时候该简单”的判断。

### 35. 这个系统最关键的 Agent 能力是什么？

我觉得最关键的是“在不确定环境下，基于工具和反馈继续推进任务”。Researcher 会根据当前结果继续检索，Reviewer 会根据评分决定是否修订，这两点都体现了 Agent 不只是调用模型，而是在有反馈的闭环里行动。

### 36. 如果你要把这个项目升级成更强的 Agent 系统，你会怎么做？

我会优先把 Researcher 并行化，让多个子问题同时搜索；然后再考虑让 Planner 和 Reviewer 提供更细粒度反馈；最后再考虑把不同 Agent 的策略做得更独立。也就是说，我会先增强执行效率和闭环质量，而不是一开始就把系统做得特别复杂。

### 37. 你这个系统里真正体现“自主性”的地方在哪里？

我觉得主要体现在两点。第一，Researcher 会根据当前结果继续检索和改写 query；第二，Reviewer 会根据评分结果决定是否带着反馈重新规划和补检索。也就是说，它不是一条完全死板的流水线，而是在边界内根据中间结果做判断。

### 38. 你为什么没有做 Agent 之间自然语言消息通信？

因为对这个项目规模来说，统一数据结构比自然语言消息总线更清晰也更稳。子问题、证据项、草稿和审查结果都可以直接结构化传递，这样更容易调试，也更容易保证上下游接口清楚。面试里这反而能体现我在复杂度控制上的判断。

### 39. 如果让你用一句话概括这个项目的 Agent 设计哲学，你会怎么说？

我会说：先把 Agent 做成可控工作流，再逐步增加自主性，而不是一开始就追求完全自治。因为对大多数真实系统来说，可解释、可评测、可终止通常比“看起来更智能”更重要。

## 四、跨岗位高频追问

### 40. 为什么不用 LangChain / Dify 之类的框架？

因为这个项目的规模其实不大，我更希望直接展示自己对状态流转、工具调用、结构化输出和调试链路的理解。如果一开始就上重框架，虽然搭得快，但很多核心细节会被框架抽象掉，反而不利于面试展示。

### 41. 这个项目里你最满意的设计是什么？

我最满意的是“引用约束 + Trace + 运行工件”这三个点结合起来。因为它们共同解决了一个很关键的问题：不仅让系统能输出结果，还让这个结果能被检查、被解释、被复盘。我觉得这比单纯多做几个功能更有工程价值。

### 42. 这个项目里你觉得最薄弱的点是什么？

如果要实话实说，我觉得最薄弱的点是评测规模还不够大，知识库规模也还偏小。现在这个版本更偏原型系统而不是生产系统，所以我优先保证了完整链路、可解释性和可演示性。后续如果继续做，我会优先补更大规模的评测和更真实的数据集。

### 43. 你怎么证明这不是一个“套壳项目”？

我会从三个角度证明。第一，它不是单次调用模型，而是完整工作流；第二，它有工具调用、失败恢复、引用约束和 Reviewer 闭环；第三，它有 Trace、运行工件和评测脚本。这些东西加起来，已经不是单纯把模型 API 包一层页面了。

### 44. 你觉得这个项目最适合用来展示你哪方面的能力？

我觉得最适合展示的是“把大模型能力变成一个可控系统”的能力。包括工作流拆分、RAG 设计、工具调用、质量控制、可观测性和评测意识。这几个点正好是大模型开发和 Agent 应用岗位都很看重的部分。

### 45. 如果只能让你用 1 分钟介绍这个项目，你会怎么说？

我会说：这是一个可验证的研究型 Agent 工作流系统。它能把研究主题拆成子问题，自动搜索和检索证据，生成带引用的结构化报告，并通过 Reviewer 做质量检查。相比普通问答系统，它更强调结果可信、过程可追踪、效果可量化。

### 46. 如果面试官追问“你这里最容易被质疑的点是什么”，你怎么答？

我会主动说两点。第一，当前评测规模还不算大，所以我不会夸大实验结论；第二，默认模式下为了保证可运行性，有些模块用了轻量 fallback，而不是一开始就依赖重型本地模型。但这两个点我都会如实说明，因为它们属于项目边界，不是逻辑漏洞。

### 47. 你在这个项目里体现出的工程思维是什么？

我觉得主要有三点。第一是先做完整链路，再逐步增强单点模块；第二是所有容易失控的地方都加边界；第三是每个重要设计都尽量留证据，比如 Trace、运行工件和评测摘要。这样项目不仅能跑，还能解释和复盘。

### 48. 如果让你重新做一次，你会坚持保留哪三个设计？

我会保留三点：workflow 主链路、引用约束、Trace 与工件保存。因为这三点决定了这个系统是不是一个“可信、可解释”的研究助手。至于模型、检索算法或者 UI 细节，都可以继续优化，但这三个设计是系统价值的核心。

### 49. 这个项目最容易打动面试官的地方是什么？

我觉得不是功能数量，而是它形成了一个完整闭环：有输入、有过程、有证据、有输出、有评测、有复盘材料。很多项目只展示最后结果，但这个项目能把“为什么得到这个结果”也展示出来，这一点更容易建立说服力。

### 50. 如果面试官让你现场改一个方向，你最有把握怎么展开？

我最有把握从三个方向展开：第一，增强检索质量，比如切到真实 embedding/reranker；第二，增强 Agent 执行效率，比如并行化 Researcher；第三，增强评测体系，比如增加更大规模 eval 集和更细粒度指标。这三个方向都和现有架构兼容，扩展路径比较清楚。

### 51. 你这个项目最大的架构取舍是什么？

我觉得最大的取舍是：我没有把它做成分布式服务，也没有做成完全自治多智能体，而是保留为一个单体、可控的 workflow 系统。这样做牺牲了一部分“架构想象空间”，但换来了更低的复杂度、更高的可调试性，以及更适合面试展示的清晰链路。对这个阶段来说，我认为这是更合理的选择。

### 52. 为什么你没有一开始就做成 FastAPI + 前后端分离？

因为这个项目当前最重要的目标不是上线服务，而是验证链路、展示能力和支持答辩。Streamlit 足够承担演示入口，CLI 也能支持快速验证。等核心链路稳定后，如果真要走产品化，再拆成 API 服务会更自然。

### 53. 你的评测体系有哪些局限？

我会主动承认两点局限。第一，当前 eval 集规模还不够大，更多是小规模验证；第二，现在的指标更偏检索和流程层面，还没有做到特别细粒度的事实一致性评测。所以这些结果能说明设计方向有效，但我不会把它包装成“已经达到生产级评测标准”。

### 54. 如果评测指标很好，但用户仍然觉得答案不靠谱，你会怎么看？

这说明“指标好”不等于“用户信任高”。比如检索命中了正确文档，不代表最终报告表达就一定让人放心。所以我才会把引用、Trace 和运行工件也一起做出来。对这种系统来说，用户信任来自结果、证据和过程三者一起成立，而不只是某个指标漂亮。

### 55. 你怎么理解这个项目里的 fallback 设计价值？

fallback 的价值是把系统从“理想条件下能跑”变成“条件不完整时也能完成任务”。比如没有 API Key 时还能走启发式模式，Web Search 出问题时知识库还能兜底。这种设计能体现工程系统思维：不是只追求最好情况，而是考虑坏情况怎么活下来。

### 56. 如果 Web Search 返回了低质量甚至冲突信息，你怎么处理？

当前版本的处理思路比较务实：一是保留来源，避免把结果混成黑盒结论；二是用知识库结果做相对稳定的补充；三是 Reviewer 和引用约束会把明显无支撑的部分压住。更进一步如果继续做，我会加强 source quality 的评分和冲突证据显式展示。

### 57. 你为什么没有把 Reviewer 做成完全基于大模型的“第二个评审智能体”？

因为当前阶段我更看重可解释性和稳定性。用规则化评分虽然没那么“炫”，但好处是清楚、可控、容易复现，也方便在面试里说明为什么一篇报告被判通过或不通过。现在它还能把这些反馈继续传回前面的规划和检索阶段，先把闭环做实，再考虑后续把规则和 LLM 审查结合起来。

### 58. 你的项目如果数据规模扩大十倍，最可能先出问题的地方是什么？

我觉得最先出问题的会是检索效率和评测维护成本。因为知识库一大，召回、精排和实验都要重新考虑性能与成本；同时，人工维护 eval 集也会变难。所以如果规模继续扩大，我会优先引入缓存、索引优化和更系统化的评测数据管理。

### 59. 这个项目在安全或风险控制上你考虑了什么？

当前项目不是生产系统，所以没有做到完整安全体系，但我考虑了几个基础点。第一，不让系统无限循环；第二，保留运行痕迹便于排查异常；第三，把外部搜索和本地知识库区分开，避免所有结果都被无条件信任。也就是说，我先从“系统别失控、结果别黑盒”这两个最核心风险入手。

### 60. 如果面试官最后问你：这个项目最能说明你和别人的区别是什么？

我会说，区别不在于我用了多少模型或多少 Agent，而在于我把一个研究型任务做成了完整工程闭环：有拆解、有检索、有写作、有审查、有引用、有评测、有工件。很多项目停留在“能出结果”，但这个项目更强调“为什么出这个结果、怎么验证这个结果”。
